# v0.2 plan: strings + a very small gc

status: thinking out loud. not implemented yet. i'm writing this the same
day v0.1.0 shipped because the moment the readme was live i started
wanting `print "hello"` to work. no code here matches reality yet.

## what 0.2 should let you write

```
fn main() {
    let name = "world";
    let greeting = "hello, " + name;
    print(greeting);
    print(len(greeting));
    return 0;
}
```

concrete feature list:

- string literals with `"..."` syntax, plus the escape sequences i
  actually use: `\n`, `\t`, `\"`, `\\`. no unicode escapes in 0.2.
- `+` overloaded to concatenate two strings.
- `len(s)` as a builtin (like `print`).
- strings are values, comparable with `==` / `!=`.
- `print(s)` prints the string without quotes and without a trailing
  newline beyond what the string contains.

not in 0.2: substring, indexing, splitting, formatting, interpolation.
those can wait until the value representation survives a few weeks of
me actually writing programs with strings.

## value representation

today a runtime value is a single `i64`. that's going to have to change.
two options:

### option A: sum-type `Value { Int(i64), Str(StrPtr) }`

16-byte value: 1-byte discriminant + 8-byte payload + padding. easy to
think about. the interp stays readable. the jit gets worse: every push
and pop doubles in size, every arithmetic op has to type-check, and the
val stack's nice `mov [r15 + r14*8], rax` story becomes a 16-byte
store.

### option B: tagged i64 (low-bit tag)

keep the value as a single 64-bit slot. use the low bit as a type tag:
- `low-bit = 0` → the upper 63 bits are an `i64` shifted left by 1
- `low-bit = 1` → the upper 63 bits are a pointer, right-shift by 1
  to get the actual heap address (heap allocations are 2-aligned so
  the low bit is always zero before shifting)

this is what ocaml does. pros: val stack stays 8 bytes per slot, all
arithmetic stays fast, interp and jit both keep the same storage shape.
cons: i need to mask/unmask every time i look at the value, arithmetic
on pointers needs shifts, and the int range drops from 64 to 63 bits.

i think B is obviously right. the whole point of the project is that
the jit stays simple. doubling the val-stack slot width would ripple
into every single template and double the memory traffic in the hot
loop. the 1-bit shift on every int op is free on x86-64 — `imul` runs
at 1 cycle throughput; one extra `shl` is noise.

### option C: nan-boxing

use the IEEE-754 nan payload to stash tagged pointers inside a float.
elegant, but i don't have floats, so i'd be using a float-shaped value
as a carrier for not-floats, which is more confusing than B.

**decision: B.** write it up in `docs/value-rep.md` before i touch code.

## heap layout

one growing arena per compiled program. in v0.2 the arena is just a
`Vec<u8>` in rust with a bump pointer, handed to the jit via a new
helper reg. string layout:

```
struct HeapStr {
    len:   u32,   # byte length, not codepoint length
    hash:  u32,   # fnv-1a, for later dict keys
    bytes: [u8; len]
}
```

8-byte header, then the bytes. alignment: allocate at 8-byte boundaries
so the low 3 bits of a heap pointer are always 000 and the tag-bit
scheme from option B works even if i later add a second tagged type
that steals the next bit up.

the jit doesn't touch the allocator directly. it calls back into a
`jit_alloc(nbytes: u32) -> *mut u8` helper in rust, same pattern as
`jit_print`. the allocator is slow; i don't care yet. the fast paths in
the tight loops are int arithmetic and local loads, which don't
allocate. i measure before i optimize.

## the gc

**strategy: stop-the-world mark-and-sweep.** tricolor is overkill at
this scale. concurrent gc is definitely overkill. i want:

1. trigger on allocation when arena size crosses a threshold (start at
   128kb, double on each collection).
2. roots: the val stack from `[r15]` to `[r15 + r14*8]`, plus the args
   and locals of every live frame (i.e. `[r15 + r13_k*8]` for each
   frame `k` on the call chain).
3. mark: walk every root, for each one check the low-bit tag, if it's
   a string pointer scan the header-byte it references (no internal
   pointers yet) and set a mark bit.
4. sweep: walk the arena, move live objects to a new arena (copying
   collector is simpler than sweeping with a free list because strings
   are variable-size and i don't want a freelist allocator yet).
5. update pointers: for each root holding a stale string pointer, look
   up the forwarding address the copy phase wrote into the old header
   and rewrite the root in place.

### walking the frames

this is the part that makes me nervous. the jit doesn't currently leave
enough breadcrumbs to walk the call stack at an arbitrary point. the
compile-time info i need:

- for each call instruction's return-address, the number of active val
  stack slots (which i already know at emit time from the ir level).
- for each frame, the `r13_k` value at entry (= saved r13 on the
  machine stack).

plan: **emit a stackmap table alongside the code**. a parallel
`Vec<(return_pc, live_slot_count, frame_base_offset)>` that the gc
consults at collection time. the jit already records
`op_offsets[i]` per op; extending this to record a stackmap row per
`Op::Call` (and per `Op::Alloc`-alike, once i have one) is a few dozen
lines.

the alternative is a shadow stack maintained by push/pop instructions
at every frame entry / call site. more runtime cost, less compile-time
bookkeeping. i'll prototype the stackmap approach first and fall back
to a shadow stack if the accounting gets ugly.

### triggering collection safely

the gc must only run when the val stack is in a known-good state. "in
a known state" means: between ops, not mid-op. which means: the
allocator helper is the only place that triggers. `jit_alloc` checks
the threshold, collects if needed, then allocates. since the allocator
is reached through a normal sysv64 call from jit code, rsp is 16-aligned
and the caller-saved regs (which hold nothing the gc cares about) are
free to clobber. callee-saved r13/r14/r15 are preserved — my gc reads
r14 as it was before the call.

actually wait: between ops r14 points one slot past the last live
value, so `[r15 + 0]` through `[r15 + (r14 - 1) * 8]` is the live val
stack. the top-1 semantics are easy to forget.

this whole thing rests on "gc only runs when r14 and r13 are accurate
and no in-flight op has a value sitting in rax that isn't on the val
stack yet". the invariant i need to preserve is: every allocation site
in the ir emits a push or a call, not a raw arithmetic op. which is
already true.

## jit impact

concretely, the encoder gains:

- `Op::Str(str_id)` ir op. at emit time, `str_id` looks up the
  interned literal string, the interp and jit pre-hand a preallocated
  `HeapStr` for each literal into the arena on program load. the op
  itself just pushes the tagged pointer.
- `Op::Concat` ir op. emits a call to `jit_str_concat(a, b) -> ptr`,
  which allocates a new `HeapStr` of combined length, memcpys the two
  payloads in, and returns the pointer. the templated version later
  can do a small-string inline optimization; not for 0.2.
- `Op::Add` grows a runtime type check: peek both operands, if both are
  ints do `add rax, rcx`, if both are pointers dispatch to a string
  concat helper, otherwise throw. this is the first op whose shape
  isn't fixed at compile time. i'd rather specialize at compile time
  via a type inference pass, but that's v0.3+.

the hot path (`fib`, `collatz`, `gcd`) has no strings and no allocs, so
the existing micro-benchmarks should stay within a few percent of
today's numbers. that's the line i don't want to cross.

## open questions

- interning: do i intern string literals but not constructed strings?
  probably yes. interned literals can live outside the gc arena, which
  simplifies their lifetime (they're static for the program's life).
- should `==` on strings be pointer-equality (fast, requires full
  interning) or content-equality (right behavior but slower)? decision:
  content-equality via the precomputed `hash` field as a fast reject,
  then memcmp. don't force interning on constructed strings.
- at what arena size should i double? guess 2x, but i should measure
  against the 1000-concat stress test before committing.
